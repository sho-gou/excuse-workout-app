# README

## 1. サービス概要
- **このサービスはどんなサービスか**
  完璧にやろうとするあまり、1日サボったことでアプリを開かなくなり、そのまま習慣自体をやめてしまう人のための、「サボった言い訳をあえて記録・可視化して、アプリからの離脱を防ぐ」**挫折後の復帰支援サービス**です。
- **誰の、どんな課題を、どう解決するか**
  「今度こそ本気で続けたい」と高い理想を掲げて始めるものの、急な残業や付き合いで1日サボった際、強い罪悪感からアプリを開く心理的ハードルに負け、そこから**芋づる式**に完全挫折（フェードアウト）してしまう人の課題を解決します。筋トレの成果だけでなく「サボった言い訳」も1タップでスタンプ感覚で記録できる受け皿を用意することで、サボった日もアプリとの接点を維持し、翌日以降の行動再開（リスタート）を心理的にサポートします。

## 2. このアイデアはどこから生まれたか
自身が過去に何度も筋トレに挑戦するも、すべて「1日サボったこと」をきっかけにフェードアウトしてしまい、どうしても継続できない強い劣等感や悔しさから生まれました。

### 2-1. きっかけとなった体験・感情
健康診断の結果を見て、自分の体型の変化や将来の健康リスクにショックを受け、「今度こそ本気で筋トレを頑張るぞ」と決意してスケジュールを完璧に立てました。しかし、いざ開始すると「今日は仕事が遅かったから」「急な飲み会が入ったから」と、その日の忙しさを言い訳にして1日サボってしまいました。翌日、既存の筋トレアプリを開こうとした際、カレンダーにできた「空白」と「昨日サボってしまった」という強い罪悪感から、アプリを開くこと自体が億劫になり、そこから雪だるま式にやらなくなって挫折してしまいました。その度に「また続けられなかった」「このままでは自分に甘い、だらしないお父さんになってしまう」という強い自己嫌悪と焦りを抱いていました。

### 2-2. なぜそれが「気になった」のか
30代を過ぎ、年齢的にもここで体型や健康管理を諦めたら、本当に年齢を言い訳にする「だらしない大人（父親）」になってしまうという強い危機感があったからです。また、自分の性格として「やると決めた時の理想は高い（スケジュールを完璧に組む）」のに、「いざ実行フェーズで1日でも狂うと、罪悪感からアプリごと投げ出してフェードアウトしてしまう」という、完璧主義ゆえの離脱メカニズム（理想と現実のギャップ）に強い違和感と課題感を持っていたためです。

## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと
- 過去の服が入らなくなる、見た目が悪くなる
- 体力の低下を感じる
- 筋トレのスケジュールを立てても長続きしない（三日坊主のループ）

### 3-2. 本当に解決したい課題は何か
- なぜ筋トレが続かないのか？ ➡️ 一度サボるとカレンダーに空白ができ、「自分はダメだ」という罪悪感からアプリを開くモチベーション自体が薄れてしまうから。
- なぜ開かなくなると困るのか？ ➡️ 「サボった日の受け皿」がないため、1日の挫折がそのままアプリの離脱、ひいては習慣化の完全な終了（フェードアウト）に直結してしまうから。

**➡️ 本当に解決したい課題：**
「目先の忙しさや付き合いで1日サボった際、強い罪悪感やアプリを開く心理的ハードルに負けてしまい、行動を再開できずにそのまま**芋づる式に完全挫折（フェードアウト）**してしまうこと（＝サボった後の復帰支援の欠如）」

### 3-3. なぜ「言い訳の記録」が行動再開（復帰）につながるのか
サボった時にユーザーの心の中では、「誰かに焚き付けて（励まして）ほしい」という気持ちと、「自分のことなんだから自分で管理すべきだ」という自立の気持ちが葛藤しています。本アプリは、言い訳をあえて記録・可視化させることで、この葛藤を解消し自発的な復帰を促します。

1. **罪悪感の緩和と客観視（メタ認知）**
   サボった理由を言語化（スタンプ選択）してアプリに記録することで、「自分がダメだからサボった」という主観的な罪悪感を、「外因によるものだ」と客観視することができます。これにより、翌日にアプリを開く心理的ハードルが下がり、アプリとの接点が維持されます。
2. **自己解決（セルフコントロール）の誘発**
   記録された言い訳を自分で見返すことで、「とはいえ、これはただの甘えだったな」「明日は時間が取れるな」と、ユーザー自身が客観的なデータに基づいて自分を焚き付け、自発的に行動を再開（リスタート）するきっかけを生み出します。

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー
- **年齢設定：** 30代後半〜40代前半の男性（特に既婚・子持ちのビジネスパーソン）
  - **理由：** 代謝が落ちて体型の崩れや健康診断の結果がリアルに気になり出す年代であり、同時に「子供や家族にとってカッコいい父親でありたい」という動機が生まれやすい世代だからです。
- **生活・状況設定：** 残業や突発的な飲み会、休日出勤など、自分の意思だけでは時間をコントロールしにくいサラリーマン。
  - **理由：** 時間が潤沢にある学生などとは違い、「忙しさ」という強力かつ正当な「サボる言い訳」が日常的に発生するため、本アプリの「言い訳を逆手に取って継続させる仕組み」が最も必要とされるからです。また、高い理想（完璧な計画）を掲げがちだからこそ、**一度計画が狂うと完璧主義が災いして芋づる式に自滅してしまうという挫折（三日坊主）のメカニズムを現在進行形で繰り返している層**でもあります。
- **環境設定：** 基本は自宅での自重トレーニング（スクワット、腕立て伏せなど）、または通勤中などの隙間時間。
  - **理由：** ジムに通う時間的・心理的ハードルが高い環境だからこそ、自宅で「1分でもやればOK」と思える超低ハードルな受け皿をアプリ側で用意し、実行のハードルを解消します。やれなかった日にも「ジムに行けなかった」ではなく、「家でやる時間が作れなかった言い訳（子どもの寝かしつけで寝落ちした、など）」を1タップでアプリに受け止めさせ、離脱を防ぐ必要があるからです。

### 4-2. 自分とユーザーの距離
- **自分自身**
  - **理由：** 自分自身が「仕事の忙しさ」を理由にして、何十回も筋トレを挫折し、その度にアプリを開かなくなっていった当事者だからです。ユーザーがどの瞬間に誘惑に負け、どんな罪悪感を抱き、なぜアプリを開かなくなるのかが痛いほどリアルに理解できるため、この距離感を選びました。

### 4-3. 実際に使われる可能性について(需要・存在確認)
- **まず最初に使ってくれそうな人は誰か？**
  自分と同じように、健康診断の結果に焦りを感じて筋トレアプリを始めたものの、残業や飲み会を理由に数日記録が途切れた瞬間に、完璧なカレンダーが崩れたショックでアプリを放置・アンインストールしてしまった経験のあるサラリーマン。
- **その人は、どんな場面でこのアプリを思い出すか？**
  夜、仕事が終わってクタクタで帰宅した時、または飲み会から帰ってきたベッドの中で、「あ、今日も筋トレやれなかったな…」と罪悪感を覚えた瞬間。
- **今の生活の中で、代わりに使っているものは何か？**
  - 有名な既存の筋トレ管理アプリ（またはスマホのカレンダーや手帳への手書き記録）
- **それを置き換えてまで使う理由はあるか？**
  既存之ツールはすべて「頑張ってやったこと（100点の日）」を記録・賞賛するものであり、サボった日はただの「空白（0点）」になり離脱を加速させます。本アプリは、サボった日に「仕事が忙しかった」「飲み会だった」という言い訳を1タップで記録でき、サボった日も10点のログとして受け入れる唯一のサービスだからです。「サボった日にアプリを開けなくなる人のための復帰支援」という体験価値は、他のツールでは替えが利きません。

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ(導入・継続の流れ)
- **どんな場面・困りごとがきっかけで、このサービスを知る・思い出すか**
  健康診断の再検査通知が届いた時や、久しぶりに着たスーツのズボンがきつくなっていた時。「今度こそ本気で続けたい、でもまた数日サボっただけでアプリを開かなくなる（芋づる式の挫折）のが怖い」とネットで「筋トレ 続かない 罪悪感」「三日坊主 アプリ 挫折」などを検索したことがきっかけで知ります。
- **最初の一歩（登録・導入）で、ユーザーは何をする必要があるか**
  1. Google認証などのSNSログインで10秒でアカウント作成。
  2. 最初に見る画面で、自分の目標（例：マイナス5kg、カッコいい父親になる）を設定。
  3. 「これだけは毎日やる」という超低ハードルなメニュー（例：スクワット10回）を1つだけ選択する。（※導入のハードルを下げるため、細かい体重入力や複雑なスケジュール設定はあえてさせません）
- **利用を続ける理由は何か？**
  - **アプリを開く心理的ハードルの低さ：** 疲れて帰ってきても、「やる」か「言い訳を選んでサボる」のどちらか1タップで今日のログが終わるため、サボった日であってもアプリを開くこと自体の心理的負担（気まずさ）がありません。
  - **カレンダーの接点維持効果：** カレンダーが空白にならず、「残業」「飲み会」といった言い訳スタンプで埋まっていくためアプリとの接点が途切れません。これを見返すことで「自分はこんなに言い訳をしていたのか、今日は5回だけでもやって青（達成）にしよう」という自発的な復帰が生まれます。
  - **同世代とのゆるい連帯感（※本リリース以降の想定）：** 自分と同じように「仕事が忙しくてサボりそうになっているサラリーマン」が、言い訳しながらもなんとかアプリに生存しているタイムラインを見ることで、「自分だけじゃない、また明日から再開しよう」とモチベーションが復帰します。

### 5-2. それでも自分が作りたい理由（なぜ「言い訳記録」が必要なのか）

上記の通り、既存のサービスはどれも「やる気がある時（100点の日）」の成果を高めるための素晴らしいツールです。しかし、これらは**仕事や付き合いで毎日ヘトヘトになりながら働くユーザーが「どうしてもやれなかった日（0点の日）」に、完璧主義が災いして自滅してしまう心理をすくい切れていない**という“ズレ”があります。ここに、私が本アプリを作りたい強い理由があります。

- **相互監視（みんチャレ）の限界：**
  サボった日に「自分だけ何も投稿していない」という状態が可視化されるため、チームへの申し訳なさや強い罪悪感が生じ、それが引き金となってアプリを開かなくなり（フェードアウト）離脱してしまいます。
- **動画レッスン（自宅トレーニング）の限界：**
  1回10〜30分のまとまった拘束時間が発生するため、夜遅く帰宅したヘトヘトな日には起動する心理的ハードルが高すぎます。さらに、一度サボると積み上げた綺麗なカレンダーの連続達成記録がリセットされるため、完璧主義な人ほどモチベーションが崩壊し、芋づる式の離脱に繋がります。
- **数値管理（筋トレmemo）の限界：**
  細かな重量や回数の入力が必要なため、疲れた脳にはその入力作業自体が負担になります。また、「やった実績」しか残せないため、サボった日はただの「空白」となり、存在を忘れてフェードアウトしていきます。

#### 結論：だからこそ「言い訳記録」が必要である
既存アプリはすべて**「サボったらそこで記録が途切れ、アプリとの接点が消える仕様（空白＝悪）」**になっています。
本当に必要なのは、サボった日を「失敗」として突き放すことではなく、**「言い訳」という形で1タップで肯定的にアプリに受け入れ、アプリとの接点を繋ぎ止めること**です。
「サボったのは自分の意志が弱かったからではなく、正当な理由（残業・飲み会）があったんだ」と言語化させて罪悪感を和らげ（メタ認知）、翌日にアプリを開く心理的ハードルを下げる。このサボった後の「復帰支援の受け皿」を持つサービスは他には存在しないため、私はこのアプリを開発したいです。

### 5-3. 差別化を一文で決める
既存のアプリが「やる気がある時の成果を高めるサービス」であるのに対し、本アプリは**「サボった日にアプリを開けなくなる人のための復帰支援サービス」**です。

## 6. このサービスで提供したい価値

### 6-1. ユーザーの変化
このサービスを使うことで、ユーザーは以下のように変化します。
- **行動の変化：** 工作や飲み会で疲れて帰ってきた日でも、アプリを開いて「1タップでサボった言い訳を記録する」という行動が維持されます。アプリとの接点が途切れないため、翌日以降に「今日は少しだけやって復帰しよう」と、再び行動を起こせるようになります。
- **気持ちの変化：** サボってしまった日も「言い訳の記録」という形でアプリに受け入れられるため、挫折に伴う過度な自己嫌悪（「どうせ自分は三日坊主だ」という諦め）が和らぎ、「明日またリスタートしよう」と前向きに捉えられるようになります。
- **考え方の変化：** 「完璧なスケジュールをこなすこと」だけが継続ではなく、「サボりながらも、自分の言い訳を自覚しつつ、アプリを開き続けて完全に辞めずに生存し続けること」こそが真の継続・復帰であると気づくことができます。

### 6-2. 価値を一文で表す
「このサービスは【いつも忙しさを言い訳に筋トレアプリを挫折してしまう人】が、【サボった日も言い訳を記録することでアプリとの接点を維持し、罪悪感を乗り越えて何度でも復帰できるようになる】サービス」

## 7. このアプリで実現すること

プログラミング初学者として、まずは最速で「サービスの核（サボった日の接点維持と復帰率向上）」を検証できる最小限の機能（MVP）から実装し、その後ユーザーの反応を見て発展させます。

### 💡 MVPで検証したい仮説
1. **「言い訳を1タップで記録（アプリに生存）したユーザーは、何も記録しなかった（カレンダーが空白のままの）ユーザーに比べて、翌日以降のアプリ復帰率が高い（芋づる式の完全離脱が減る）か」**
2. **サボった日も『言い訳スタンプ』で簡単に記録できる受け皿があることで、完璧な計画が崩れたことによる心理的挫折（フェードアウト）を防げるか**

### MVPで作る機能
1. **ユーザー管理機能**
   - ユーザー登録、ログイン、ログアウト（Sorceryを用いたメールアドレスまたはGoogle認証ログイン）
   - プロフィール編集（目標の設定、超低ハードルなメニューの選択）
2. **日々の接点維持（実行 / 言い訳）機能**
   - 1日1回、カレンダー上の「今日の枠」または専用ボタンから1タップで記録。
   - 「実行」ボタン：やったメニューを選択して登録（DB上には `status: "done"` として保存）。
   - 「サボる」ボタン：あらかじめ用意された言い訳スタンプ「残業」「飲み会」「体調不良」「なんとなく」等を選択して登録（DB上には `status: "excuse"` として保存）。
3. **言い訳カレンダー機能（ダッシュボード）**
   - 記録結果をカレンダー上で一覧表示（Gem `simple_calendar` を想定）。
   - 「実行した日」は青、「サボった日」は赤（または選んだ言い訳のスタンプアイコン）で視覚的に一目でわかるように表示し、カレンダーの「空白（未ログイン）」を作らせないUI。

### 本リリースで作る機能
1. **サボり連動型アバター変化機能**
   - ※MVPでは「言い訳の記録による接点維持・復帰率向上」の検証に集中するため、また、キャラクターの悪化（戒め）がユーザーに過度なプレッシャーを与えて復帰を阻害するリスクを避けるため、MVP検証後の行動変容を加速させる拡張機能として位置づけます。
2. **言い訳分析（グラフ化）機能**
   - 今月自分がどの言い訳（残業、飲み会など）を何回使ったかを円グラフ等で表示し、自分のサボり傾向（トリガー）を客観的に自覚させるダッシュボード。
3. **タイムライン機能（SNS連携・ゆるい連帯感）**
   - ユーザー同士が「今日も残業でサボりました…」「出張だけどスクワットだけやリスタートしました！」とお互いの生存報告（言い訳含む）を共有し、匿名で「お疲れ！」スタンプを押し合える復帰応援コミュニティ機能。
4. **通知（リマインド）機能**
   - 夜の指定した時間に、LINE Bot等を通じて「今日の生存報告（実行または言い訳）はもう済んだ？」と通知を飛ばし、サボった日ほどアプリの立ち上げを促す機能。

## 8. このアプリの懸念点とその対策

### 懸念点①
- **何が問題になりそうか：** 「サボった言い訳を記録する」という仕様上、ユーザーがサボることに慣れてしまい、最終的に筋トレ自体を全くやらなくなってしまう（目的のすり替わり）。
- **理由・背景：** アプリを開いて言い訳をタップするだけで「記録した満足感（アプリへの生存）」が得られてしまうため、本来の目的である「筋トレによる健康管理」へ復帰しない可能性があるからです。
- **対策：** MVP検証によって「アプリとの接点が維持されれば復帰率が上がる」ことが確認できたら、本リリースにて「アバターの体型変化（視覚的危機感の付与）」や「言い訳の統計分析」を実装します。サボり（言い訳の記録）が積み重なると、マイページのアバターが「ビール腹のだらしない体型」へリアルタイムに変化していく仕様にすることで、「このままではダメなお父さんになってしまう」という危機感をユーモラスに刺激し、ユーザーが客観的に自分を自覚して再び「実行」ボタンを押したくなる行動（復帰）を促します。

## 9. 今後の展開・発展の方向性
- **ユーザーはなぜ使い続ける？：** 単なるカレンダー埋めにとどまらず、長く使うほど「自分専用のサボりトリガー分析ツール」としての価値が高まるためです。
- **データが溜まると何ができる？：** 蓄積された言い訳データから、「自分は毎週木曜日に残業でサボりやすい」といった、自身のサボりパターンが統計データとして見えてきます。これが可視化されることで、ユーザーは「木曜日にサボったのはいつものパターン（想定内）だから、金曜にリカバーすればいい」と、サボりを受け入れた上での持続可能なマイスケジュールを設計できるようになり、長期的な復帰・継続へと繋がります。
- **機能以外の工夫：** 初学者の個人開発として運用コストを抑えるため、最初は複雑なサーバー維持費がかからない構成にし、SNS（Xなど）で「#サラリーマンの言い訳筋トレ」といったハッシュタグで開発の裏側を発信することで、初期のユーザー（同じ悩みを持つお父さんサラリーマン）を巻き込んでコミュニティを作ります。

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術
- **フレームワーク：** Ruby on Rails 7.x
- **DB：** MySQL
- **デプロイ先：** AWS (EC2 / Nginx)
- **主要な使用予定ライブラリ（Gem）：**
  - `sorcery`（認証機能）
  - `simple_calendar`（カレンダー表示）
  - `Tailwind CSS`（フロントエンドのデザインUI調整）

### 10-2. 技術選定の理由
- **なぜこの技術を使うのか：** 基礎を深く学習した「Ruby on Rails」および「MySQL」をベースにすることで、実装のバグを最小限に抑え、サービスのコアロジック（実行とサボりのCRUD処理、カレンダーへの出し分け）の実装に集中するためです。
- **今回チャレンジしたい点：** `simple_calendar` の標準機能を利用し、カレンダーのマス目内に保存された `status`（done / excuse）に応じた条件分岐を組み込み、直感的に自分の接点維持状況がわかるUI/UXを構築すること。
- **不安な点：** カレンダー画面の中で、日付ごとに「実行データ」と「言い訳データ」を綺麗に出し分けるコントローラーとビューの条件分岐ロジック。これに関しては、事前に小さなモック（試作品）を作って検証します。

### 10-3. キャッチアップ不足の懸念
最近モダンなフロントエンドとして注目されている「React」や「Next.js」について、最初から導入するとキャッチアップに多大な時間を取られ、MVPのリリース（完成）自体が遅れるリスクがあります。
そのため、**MVPおよび初期リリース段階ではRailsの標準機能（Hotwire/Turbo）とシンプルな条件分岐（if文）で画面表示を構築**し、アプリがしっかりとユーザーに受け入れられ、「サボった日の接点維持による復帰率向上」という技術的・サービス的な検証を済ませた後の発展形として、本リリース以降にフロントエンドのReact化を検討する方針とします。
